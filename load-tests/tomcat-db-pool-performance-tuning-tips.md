A Pragmatic Guide to Performance Tuning in Spring-Tomcat Applications
1.0 The Foundations of Concurrency and Performance
Effective performance tuning is not about randomly changing parameters until a problem disappears. It is a disciplined engineering practice rooted in understanding the fundamental principles of concurrency and queuing theory. Misconfigurations in a high-throughput system can lead to catastrophic, non-deterministic failures that are notoriously difficult to diagnose. This section establishes the theoretical groundwork necessary to make informed, data-driven tuning decisions across the application stack.
The Physics of Queuing: Applying Little's Law
Before tuning any system, it is crucial to define the performance targets. Little's Law is a simple yet powerful formula from queuing theory that provides the fundamental relationship between throughput, latency, and concurrency. It allows us to calculate the number of worker threads a system requires to meet its service-level objectives.The formula is expressed as:L = λ × WWhere:
L  = Concurrency (The average number of requests in the system at any given time, which translates to the required number of worker threads).
λ  = Arrival Rate (The number of requests arriving per second, or RPS).
W  = Average Response Time (The average time it takes to process a single request, also known as latency, in seconds).For a practical example, consider a service that must handle  500 RPS  (λ) with an average latency of  0.1 seconds  (W). Applying Little's Law:L = 500 RPS × 0.1 s = 50This calculation tells us that the system needs at least  50 concurrent worker threads  to sustain this load. Any fewer, and requests will begin to queue up, increasing latency and potentially causing timeouts. This simple formula provides the initial, evidence-based target for sizing our primary thread pool.
Why Tuning Matters: Common Concurrency Pathologies
Misconfigured multi-threaded systems are susceptible to specific, hard-to-diagnose failures under load. These "concurrency pathologies" are not simple bugs but emergent behaviors that arise from the unpredictable timing of thread execution. Understanding their mechanisms is key to preventing them.| Concurrency Issue | Core Mechanism | Primary Risk || ------ | ------ | ------ || Race Condition | Two or more threads engage in interleaved read-modify-write cycles on shared, mutable data without proper synchronization. This occurs frequently in singleton-scoped Spring beans that maintain mutable state. | Subtle data corruption, inconsistent application state, and lost updates. || Deadlock | Two or more threads are blocked indefinitely, each waiting for a resource that the other holds. This often occurs when a thread acquires a Java lock (e.g., on a service method) and then requests a database lock held by another thread, which in turn is waiting for the same Java lock. | Complete application hang and service unavailability, often requiring a restart. || Livelock | Threads are not blocked but are constantly changing state in response to each other, making no forward progress. | High CPU exhaustion with little to no useful work being done, resulting in extreme latency. || Starvation | A thread is perpetually denied access to a required resource (like CPU time or a database connection) by higher-priority or greedy threads. | Unpredictable response times and the failure of specific tasks to ever complete. || Visibility | A change made to a shared variable by one thread is not seen by other threads due to local CPU caching. | Stale data usage and infinite loops, as threads may not see updated termination flags. |
With these foundational risks in mind, we can now turn to the practical configuration of the application's entry point: the Tomcat web server.
2.0 Configuring the Tomcat HTTP Thread Pool
The Tomcat thread pool is the application's front door for handling all incoming HTTP requests. Its configuration dictates the maximum number of simultaneous requests the application can process and serves as the first line of defense against being overwhelmed by traffic. Properly sizing this pool is the first step in building a resilient system.
Anatomy of a Request: The NIO Processing Pipeline
Modern Tomcat versions use a Non-blocking I/O (NIO) connector architecture to handle connections with high efficiency. This model decouples the physical connection from the processing thread, allowing a small number of threads to manage thousands of connections. The request lifecycle involves three distinct thread groups:
Acceptor Threads : A small number of threads dedicated to a single task: listening on the server socket for new TCP connection requests. When a connection arrives, the Acceptor hands it off to a Poller thread.
Poller Threads : These threads monitor a large set of open connections for incoming data. When a Poller detects that data is ready to be read on a socket, it assigns the request to a worker thread from the main thread pool for processing.
Worker Threads : This is the primary thread pool (governed by maxThreads) that executes the application's business logic, such as Spring controllers and service methods. These threads perform the actual work of handling the request and generating a response.
Core Tuning Parameters and Heuristics
Tomcat's performance is primarily governed by four key parameters within its Connector configuration. Understanding their distinct roles is critical for stability.| Parameter | Default Value | Strategic Impact || ------ | ------ | ------ || maxThreads | 200 | Sets the maximum number of worker threads, defining the absolute limit of concurrent request processing. Each thread consumes ~1MB of native memory for its stack. || minSpareThreads | 10 | The minimum number of worker threads kept alive and ready, even when idle. This prevents latency spikes by avoiding thread creation costs during sudden traffic bursts. || maxConnections | 8192 (for NIO) | The maximum number of TCP connections the server will accept and manage. This is the final buffer before the server starts refusing connections. |
With the entry point configured, we now shift our focus from handling incoming web requests to managing the most common performance bottleneck in nearly every web application: the database connection pool.
3.0 Optimizing the HikariCP Database Connection Pool
The database connection pool is frequently the most significant bottleneck in a web application. The database is often the most rigid and expensive resource in the stack, and misconfiguring its connection pool can starve the entire system of the resources it needs to function, leading to cascading failures and poor performance.
The Sizing Paradox: Why Less Is More
A common but dangerous misconception is that a larger database connection pool leads to better performance. It is a basic Law of Computing that given a single CPU resource, executing  A  and  B  sequentially will  always  be faster than executing  A  and  B  "simultaneously" through time-slicing. Oversized connection pools force the database into exactly this kind of inefficient time-slicing, leading to increased resource contention. When hundreds of connections compete for a handful of CPU cores, the database spends more time on context switching—managing the competition between connections—than it does executing queries.An Oracle Real-World Performance group demonstration showed that by simply reducing a connection pool size, with no other changes, an application's response time improved by over  50x . This is because a smaller, saturated pool forces requests to be queued and executed by the database in a tight, efficient sequence, minimizing I/O contention and maximizing throughput.
A Formula for Optimal Sizing
While every application is unique, a widely respected, cross-database formula popularized by the PostgreSQL community provides an excellent starting point for sizing a connection pool. The original formula is connections = ((core_count × 2) + effective_spindle_count). For modern systems with fast SSD storage where effective_spindle_count is effectively 1, this simplifies to:maximumPoolSize = (Ncpu × 2) + 1In this formula, Ncpu refers to the number of  physical CPU cores  available to the database server, not including hyper-threads.For example, on a server with  4 physical cores , the calculation results in a recommended starting pool size of (4 × 2) + 1 = 9. This may seem counter-intuitively small, but it is sized to match the database's actual parallel processing capability.
Configuration for Production Stability
Beyond sizing, several HikariCP parameters are critical for ensuring the resilience and stability of the application in a production environment.
minimum-idle : This should be set  equal to  maximum-pool-size . This creates a fixed-size connection pool, preventing the overhead of creating and destroying connections during load fluctuations (a process known as connection churn).
connection-timeout : This should be set to a "fail-fast" value, typically between  2000ms and 5000ms . If the pool is exhausted, this setting prevents application threads from hanging indefinitely while waiting for a connection, allowing them to fail quickly and gracefully.
max-lifetime : This setting is crucial for preventing stale connections. It should be set to a value slightly shorter than any database or network-enforced idle timeouts. A safe value is often around  30 minutes (1800000ms) .
leak-detection-threshold : Setting this to a low value, such as  2000ms , is an invaluable diagnostic tool. If an application thread holds a connection for longer than this threshold, HikariCP will log a stack trace, immediately identifying slow queries or unclosed transactions that are starving the pool.Having tuned the pool for handling synchronous database work from web requests, we now turn our attention to managing separate, asynchronous background tasks.
4.0 Managing Asynchronous Workloads with Spring's ThreadPoolTaskExecutor
Many applications need to perform background processing—such as sending emails, processing data, or calling external APIs—using Spring's @Async annotation. These tasks run in a separate ThreadPoolTaskExecutor, which is independent of the Tomcat web thread pool. Misconfiguring this executor is a common source of high latency for background work and can lead to severe memory issues.
The Counter-Intuitive Sizing Logic
The ThreadPoolTaskExecutor uses a specific and often misunderstood logic to allocate tasks, which can lead to unexpected behavior if not configured carefully. The rules are applied in the following order:
If the number of running threads is less than corePoolSize, a new thread is created to handle the task.
Once corePoolSize is reached, all new tasks are placed into the queueCapacity.
Only if the queue is full , the executor will create a new thread to handle a task (up to the maxPoolSize limit).
If the queue is full and maxPoolSize has been reached, the task is rejected according to the configured rejection policy.A very common configuration error is to set a large maxPoolSize while leaving queueCapacity at its default of Integer.MAX_VALUE. Because the queue is effectively unbounded, it will  never fill up . As a result, the thread pool  will never grow beyond  corePoolSize , and background tasks will accumulate in the queue, leading to high processing delays and potential memory exhaustion.
Choosing a Rejection Policy
The RejectedExecutionHandler determines how the system behaves when the executor and its queue are completely saturated. Choosing the right policy is critical for system stability.
AbortPolicy (Default) : Throws a RejectedExecutionException. This is a safe default, as it immediately notifies the calling thread that the task could not be scheduled.
CallerRunsPolicy : The thread that submitted the task executes the task itself. This creates a natural form of back-pressure, as the submitting thread (e.g., a Tomcat worker thread) is slowed down, preventing it from overwhelming the async pool.
DiscardPolicy : Silently drops the new task without throwing an exception. This is dangerous and should only be used for non-essential tasks where data loss is acceptable.
DiscardOldestPolicy : Discards the task at the head of the queue to make room for the new task. This is useful in scenarios where processing only the most recent data is important.Tuning individual components in isolation is a recipe for failure. The next section explains how to approach tuning as a cohesive, system-wide strategy.
5.0 An Integrated Strategy for Holistic Tuning
Performance is an emergent property of a system, not a feature of a single component. Resource pools for web requests, database connections, and asynchronous tasks must be sized in harmony. Tuning one pool without considering the others will simply shift the bottleneck, not resolve it. A successful strategy requires a holistic approach, starting with the most constrained resource and working outwards.
5.1 Step 1: Establish the Baseline with Little's Law
Begin by using Little's Law (L = λ × W) to calculate the target concurrency level required to meet your business objectives for throughput (Requests Per Second) and latency. This number provides an evidence-based target for your Tomcat maxThreads configuration and grounds the entire tuning process in concrete performance goals.
5.2 Step 2: Anchor the System by Sizing the Database Pool First
The database connection pool is typically the most rigid resource in the stack and is constrained by physical hardware. It should be sized first to establish a firm performance ceiling for the entire system. Use the PostgreSQL-derived formula maximumPoolSize = (Cores × 2) + 1, where Cores is the number of physical CPU cores on the database server. This anchors your system's throughput to the database's maximum efficient capacity.
5.3 Step 3: Size the Tomcat Pool in Relation to the Database Pool
Once the database pool size is fixed, the Tomcat maxThreads value must be sized appropriately. A common rule of thumb for enterprise applications is Recommended_DB_Connections = maxThreads × 1.25, ensuring the database pool can support the web threads plus additional background tasks.However, for I/O-bound web applications, maxThreads can and often should be significantly larger than the number of CPU cores. The Goetz formula provides a more nuanced calculation: Nthreads = Ncpu × Ucpu × (1 + W/C), where W/C is the ratio of wait time to compute time. For a typical web application where threads spend most of their time waiting on network or database I/O, this ratio is high, justifying a larger thread pool than the number of cores alone would suggest.
5.4 Step 4: Configure the Backlog with maxConnections and acceptCount
These parameters manage the hierarchy of queues for users waiting for a free worker thread. The process is sequential:
Tomcat uses worker threads up to maxThreads to actively process requests.
If all threads are busy, Tomcat continues to accept new TCP connections up to the maxConnections limit. These connections are held by Poller threads in a waiting state but do not occupy a worker thread.
Only when maxConnections is reached does the acceptCount queue come into play. This is the OS-level backlog for TCP connection requests  before  Tomcat even accepts them. It is, quite literally, "the line waiting to get into a popular nightclub that is full."
If this final OS-level queue fills, connection requests are refused by the operating system itself. Setting acceptCount too high is dangerous, as it can cause requests to wait so long that they time out on the client side before Tomcat ever processes them.

6.0 Monitoring and Diagnosing Performance
No tuning effort is complete without validation. Once a system is configured, it must be monitored under realistic load to verify that the changes have had the desired effect and to diagnose any new performance issues that arise. This final section provides guidance on what to watch and how to interpret the signals your application provides.
Essential Metrics and JMX Beans
Both Tomcat and HikariCP expose critical performance metrics via Java Management Extensions (JMX) beans, which can be monitored with tools like JavaMelody, Prometheus, or any JMX-compliant console.
Tomcat Threads : The Catalina:type=ThreadPool MBean is essential.
currentThreadsBusy: The number of threads currently processing requests. If this is consistently near maxThreads, the system is at its limit.
currentThreadCount: The total number of threads currently in the pool (busy + idle).
maxThreads: The configured maximum limit.
Database Connections : The Catalina:type=DataSource MBean tracks the connection pool.
numActive: The number of connections currently in use. If this is consistently at the pool's maximum, the database is a bottleneck.
numIdle: The number of connections available in the pool.
Memory Usage : The java.lang:type=Memory MBean provides heap memory statistics.
HeapMemoryUsage: Tracks used, committed, and max heap size. A constant increase in used memory is a sign of a memory leak.
Analysis of Thread Dumps
A thread dump is an instantaneous snapshot of the state of every thread inside the JVM. By taking several dumps a few seconds apart, you can diagnose hangs, deadlocks, and performance hotspots. The most relevant thread states for performance analysis are:
RUNNABLE : The thread is actively executing code. A thread that remains RUNNABLE with the same stack trace across multiple dumps is likely stuck in a long-running computation or an infinite loop.
BLOCKED : The thread is waiting to acquire a lock that is held by another thread. A large number of threads in the BLOCKED state points to high lock contention or a potential deadlock.
WAITING  /  TIMED_WAITING : The thread is waiting for a signal from another thread, often because it is waiting for a resource from a pool (like a database connection) or an item from a queue.
Interpreting Log Signatures for Pool Exhaustion
Application logs often provide the first and clearest signal that a resource pool has been saturated. Different types of exhaustion produce distinct error signatures.
Tomcat Pool Exhaustion : When the maxThreads and acceptCount limits are breached, clients will typically receive  HTTP 503 Service Unavailable  responses.
HikariCP Pool Exhaustion : When the database connection pool is empty, the application will log a  java.sql.SQLTransientConnectionException  with a message like "Connection is not available, request timed out...".
System-wide Thread Leaks : When threads are created but never terminated properly, the JVM may be unable to request more threads from the operating system, resulting in a  java.lang.OutOfMemoryError: unable to create new native thread .Ultimately, building resilient and scalable applications is a cycle of evidence-based engineering: applying foundational theory, tuning components as an integrated system, and validating every change through continuous monitoring.




A Study Guide to Concurrency and Resource Management in a Spring-Tomcat Environment
Every HTTP request that arrives at your Spring-Tomcat application begins a perilous journey through a gauntlet of potential bottlenecks. The stability of your system depends not on optimizing one component in isolation, but on understanding the entire chain of resource pools—from the web server's front door to the database's back end. This guide is built on a critical principle of performance engineering: misconfiguration in one area doesn't solve a problem; it creates  cascading failures  by merely shifting the bottleneck to the next weakest link. A Tomcat pool that's too large will simply overwhelm a smaller database pool, leading to connection timeouts. A database pool that's too large will cause resource contention at the hardware level.Our mission is to arm you with a holistic mental model for tuning these components in concert. We will dissect the journey of a request, analyze the failure modes at each stage, and provide the diagnostic techniques to identify and resolve them. A mastery of this material is the difference between an application that collapses under its first significant load and one that scales gracefully and predictably.
1.0 Understanding the Landscape: Common Concurrency Pathologies
To effectively manage and tune a Spring-Tomcat application, you must first understand the fundamental concurrency problems that arise in any multi-threaded environment. Do not mistake these for abstract academic concepts. In my experience, these pathologies are the direct root cause of the most elusive and severe performance bottlenecks and system failures. A firm grasp of these issues is the foundation for building resilient, high-performance systems.The core challenges of concurrent programming can be summarized as follows:| Concurrency Issue | Core Mechanism | Primary Risk || ------ | ------ | ------ || Race Condition | Interleaved read-modify-write cycles on shared data. | Data corruption, inconsistent application state. || Deadlock | Circular resource acquisition where two or more threads block each other indefinitely. | Application hang, complete service unavailability. || Livelock | Threads are in a constant state of change but make no forward progress, often due to overly aggressive retry logic. | CPU exhaustion and high latency with no useful work being done. || Starvation | A thread is perpetually denied access to necessary resources (e.g., CPU, locks) by higher-priority or greedy threads. | Unpredictable response times and potential failure of specific tasks. || Visibility | Changes to a shared variable made by one thread are not seen by other threads due to local CPU caching. | Stale data usage, infinite loops, and unpredictable behavior. |
Analysis of Key Pathologies
Race Condition: The Lost Update Problem
A race condition occurs when the correctness of an operation depends on the unpredictable timing of multiple threads. A classic example is a simple increment operation (count++), which is not atomic. At the bytecode level, this single line of code is actually three distinct operations:
Read:  Read the current value of count from memory.
Modify:  Increment the value in a CPU register.
Write:  Write the new value back to memory.If two threads (Thread A and Thread B) attempt to increment count concurrently when its value is 10, the following sequence can occur:
Thread A reads 10.
Thread B reads 10.
Thread A computes 11 and writes it back.
Thread B computes 11 and writes it back.The final value is 11 when it should be 12. This "lost update" can lead to subtle and catastrophic data corruption in systems managing inventory, financial transactions, or user sessions.
Deadlock: The Vicious Circle
A deadlock is a terminal state where two or more threads are permanently blocked, each waiting for a resource that the other holds. For a deadlock to occur, four conditions must be met simultaneously:
Mutual Exclusion:  Resources cannot be shared.
Hold and Wait:  A thread holds at least one resource while waiting for another.
No Preemption:  Resources cannot be forcibly taken from a thread.
Circular Wait:  A chain of threads exists where each thread is waiting for a resource held by the next thread in the chain.A common scenario in a Spring-Tomcat application involves both application-level and database-level locks. Imagine this precise sequence:
Thread 1  acquires a lock on a Java object via a synchronized method, then attempts to execute an UPDATE statement on ROW_A.
Simultaneously,  Thread 2  has an open transaction that has already placed a database lock on ROW_A and now attempts to call the  same  synchronized method that Thread 1 has locked.Here, Thread 1 holds the Java lock and waits for the database lock, while Thread 2 holds the database lock and waits for the Java lock. This circular dependency creates a deadlock, freezing both threads and rendering a part of the application unresponsive.Having established this theoretical groundwork, we can now examine how these problems manifest within the first major component of the application stack: the Tomcat server.
2.0 The Front Door: Tomcat's Request Handling and Thread Pool Saturation
The Apache Tomcat Connector and its associated thread pool serve as the entry point for every user request. The configuration of this component is paramount, as it directly dictates the application's capacity to handle concurrent traffic. Misconfiguration here is a primary cause of performance degradation and service unavailability, effectively slamming the door on users before their requests can even be processed.
The NIO Request Processing Pipeline
Modern Tomcat installations use a Non-blocking I/O (NIO) connector, which employs three distinct thread groups to manage the request lifecycle efficiently. A request's journey begins when an  Acceptor  thread grabs the initial TCP connection and immediately hands it off to a  Poller . The Poller acts as a watchman, monitoring the connection for data. Only when data arrives does the Poller signal for a  Worker  thread from the main pool, ensuring that precious worker threads are only engaged when there is actual work to be done.
Critical Configuration Parameters
The capacity and behavior of the Tomcat Connector are governed by three key parameters:
maxThreads : The maximum number of worker threads that can be created to process requests simultaneously.
Default Value:  200
Purpose:  This parameter defines the absolute maximum number of concurrent requests your application can actively handle. Once all 200 threads are busy, no new requests can be processed until a thread becomes free.
maxConnections : The maximum number of simultaneous TCP connections the server will accept and maintain.
Default Value (NIO):  Varies. Common defaults are 10000 or 8192 depending on the Tomcat version and distribution. Always verify the specific version you are running.
Purpose:  This allows Tomcat to handle many more connections than active threads. Many of these connections may be in an idle "keep-alive" state, waiting for a subsequent request from a user's browser without consuming a precious worker thread.
acceptCount : The maximum length of the operating system's queue for incoming connections.
Default Value:  100
Purpose:  This queue is the final line of defense. When both the worker threads (maxThreads) and the connection slots (maxConnections) are full, the OS will queue incoming connection requests up to this limit. A useful analogy is to think of acceptCount as the  "line waiting to get into a popular nightclub that is already full."
The Path to Saturation and Failure
When the application is under heavy load, these limits are tested in a specific sequence:
Requests arrive and are assigned to available worker threads until all maxThreads (e.g., 200) are busy.
With all worker threads occupied, Tomcat continues to accept new connections up to the maxConnections limit. These connections wait for a worker thread to become available.
Once maxConnections is also reached, incoming connection requests are placed into the operating system's listen queue, which holds up to acceptCount (e.g., 100) connections.
Failure State:  When the acceptCount queue is full, the operating system can no longer accept new connections. Subsequent requests are actively refused or will time out from the client's perspective, often resulting in "Connection Refused" errors.Once a request successfully obtains a worker thread, that thread's journey is not over. It frequently needs to access back-end resources, which brings us to the next critical bottleneck: the database connection pool.
3.0 The Bottleneck: Database Connection Pooling and Blocking Calls
In most web application stacks, the database connection pool is the most common and rigid bottleneck. Establishing a database connection is a resource-intensive process, making a pool of reusable connections essential for performance. However, the size and configuration of this pool are critical. It directly determines how long application request threads will be blocked when they need to communicate with the database, and improper sizing can cripple an entire system.
The "Less is More" Paradox
A counter-intuitive but vital principle of pool sizing is that larger pools often degrade, rather than improve, performance. As documented by the creators of HikariCP and demonstrated by the Oracle Real-World Performance group, excessive connections lead to severe resource contention at the database level. When hundreds of connections compete for a limited number of CPU cores, the database spends more time on context switching (juggling tasks) and managing lock contention than on executing queries. In one dramatic demonstration, the Oracle team showed that reducing a connection pool from 2048 down to just 96 improved application response times from ~100ms to ~2ms—a greater than 50x improvement.Your primary directive when tuning a database pool is to find the smallest possible size that maximizes throughput.  Start with the formula below and tune downwards first, not upwards.
A Formula for Sizing
While every application is different, a widely respected starting point for calculating an optimal pool size is provided by the PostgreSQL project. This formula is applicable to most relational databases:connections = ((core_count * 2) + effective_spindle_count)
core_count : The number of physical CPU cores on the database server. Do not include hyper-threaded cores.
effective_spindle_count : The number of physical disk drives. For modern systems using Solid State Drives (SSDs), which have no moving parts, this value is typically simplified to 1.This formula reinforces the "less is more" principle; the optimal number of connections is directly tied to the database's physical hardware constraints (CPU cores), not the number of application threads trying to access it. For a database server with 4 CPU cores and SSD storage, the formula suggests an initial pool size of ((4 * 2) + 1) = 9.
Pool Exhaustion
When an application thread requests a database connection and the pool is exhausted (all connections are in use), the thread will block and wait. If a connection does not become available within the configured timeout, the connection pool library (like HikariCP) will throw an exception. The typical signature for this failure is a java.sql.SQLTransientConnectionException with the message:  "Connection is not available, request timed out" .While external resource pools like HikariCP manage database access, applications often rely on internal thread pools for asynchronous work, which introduces its own set of challenges.
4.0 The Background Worker: Spring's ThreadPoolTaskExecutor and Queueing Behavior
Beyond handling synchronous web requests, modern applications often need to execute background tasks, such as sending emails, processing data, or calling external services. Spring Framework facilitates this through the @Async annotation and the underlying ThreadPoolTaskExecutor. It is crucial to understand that this is a separate and distinct thread pool from Tomcat's worker pool, and its unique, and often counter-intuitive, behavior is a common source of configuration errors.
The Four-Step Thread Creation Logic
The ThreadPoolTaskExecutor does not create new threads in the way one might expect. It follows a specific four-step logic when a new task is submitted:
If the number of running threads is less than the configured corePoolSize, a new thread is created immediately to handle the task.
If the number of running threads is equal to or greater than corePoolSize, the task is  placed into the queue .
If the queue is full  AND  the number of running threads is less than maxPoolSize, a new thread is created to handle the task.
If the queue is full  AND  the number of running threads has reached maxPoolSize, the task is rejected.
The maxPoolSize Misconfiguration Trap
A critical and frequent misconfiguration arises from the default settings. The default queueCapacity for a ThreadPoolTaskExecutor is Integer.MAX_VALUE. Because this queue is effectively unbounded, it will  never fill up .According to the logic above (step #3), new threads beyond corePoolSize are only created when the queue is full. Since the default queue never fills, the pool will  never create more threads than the  corePoolSize . This renders the maxPoolSize setting completely meaningless. Developers who set a small corePoolSize and a large maxPoolSize, expecting the pool to scale up under load, will find that their asynchronous tasks pile up in a massive, memory-consuming queue, leading to extreme processing delays. Users' requests that should be handled asynchronously in seconds might instead wait for hours, all while the system  appears  to be under low load because the CPU-intensive threads defined by maxPoolSize were never created.Achieving true system stability requires a holistic approach, harmonizing the sizing of all three critical pools—Tomcat for web requests, HikariCP for database connections, and the Task Executor for background work—which is the focus of the final section.
5.0 Diagnosing Failures: Identifying Saturation and Contention
Theoretical knowledge of thread pool limits and concurrency problems must be paired with practical diagnostic skills. This section will equip you with the diagnostic skills to move from theory to practice. You will learn to read the application's "fingerprints"—its logs and thread states—to pinpoint the exact source of failure under load.

Using a Thread Dump for Analysis
A thread dump is an invaluable diagnostic tool that provides a "snapshot" of the state of every thread inside the JVM at a specific moment. For performance analysis, the most interesting thread states are:
RUNNABLE : The thread is actively executing code. A thread that remains RUNNABLE with the same stack trace across multiple dumps may be stuck in an infinite loop.
BLOCKED : The thread is waiting to acquire a lock (a synchronized block) that is currently held by another thread. A large number of BLOCKED threads indicates high lock contention.
WAITING  /  TIMED_WAITING : The thread is waiting on a condition, such as waiting for a connection from a pool or for a task to arrive in a queue.
How to Identify a Deadlock
A thread dump is the definitive way to diagnose a deadlock. When a deadlock is present, a diagnostic tool like jstack will explicitly parse the thread states and report it directly in its output with a clear message, such as:  "Found one Java-level deadlock" . The tool will then provide the details of the threads involved and the locks they are waiting for, making the circular dependency clear.
How to Identify Thread Pool Exhaustion
